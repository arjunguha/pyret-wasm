# Wiring a real `surface-parse` into the self-hosted compiler

## The target
`compile-lib`/`repl`/`desugar` call `P.surface-parse(src :: String, uri :: String) -> Program`
(an `ast.arr` `Program`, i.e. `s-program(l, _use, _provide, provided-types, provides, imports, block)`).
Today `self-compiler/compiler/parse-pyret.arr` is a STUB that `raise`s, so the real
front-end passes can't run on real source.

## How upstream does it
`pyret/lang/src/js/trove/parse-pyret.js` (1651 LOC, JS) is the real implementation:
1. runs the **JS tokenizer + GLR parser** (`pyret-base/js/pyret-tokenizer`, `pyret-parser`)
   → a **GLR parse tree** of `{name, kids, pos}` nodes (same shape as our seed's
   `src/parser/parse-core.ts` `CstNode`);
2. a big recursive tree-walk (`tr`/helpers) maps each parse-tree node to an `ast.arr`
   node via `RUNTIME.getField(ast, "s-X").app(args...)` — handling surface sugar,
   operator chains, etc. Output is the `s-program` AST.

Our seed already reuses (1) — `parse-core.ts` produces the GLR `CstNode`. The missing
piece is (2) done in a way the WASM self-hosted compiler can use (its own value model).

## Two options

### Option A — full Pyret port (pure, large)
Port the `parse-pyret.js` tree-walk to Pyret as `surface-parse`, consuming a **serialized
GLR parse tree** (conveyed into WASM, see below) and building `ast.arr` nodes with the
real constructors. Pure-Pyret, no extra JS, but ~1651 LOC of translation to port.

### Option B — host parses to a SERIALIZED AST; Pyret deserializes (recommended first)
Less porting, reuses parse-pyret.js's *logic*:
1. **Retarget** a copy of `parse-pyret.js` so that instead of `RUNTIME.getField(ast,"s-X").app(a,b)`
   it emits a plain tagged tree `{tag:"s-X", args:[a,b]}` (mechanical: the file already
   funnels through `getField(ast, NAME).app(...)` — wrap that one call site). This needs
   NO Pyret runtime in JS.
2. **Host import** (design only here; `run.ts` is owned elsewhere):
   `parse_to_ast(srcPtr, srcLen, uriPtr, uriLen) -> (astPtr, astLen)` — runs the JS
   tokenizer+GLR+retargeted-builder and writes a **flat pre-order serialization** of the
   tagged tree into WASM linear memory (reuse the `read-source`/`$str_from_mem` machinery).
3. **Pyret `surface-parse`** (this dir's skeleton, `parse-from-tree.arr`): reads the
   serialized stream and rebuilds `ast.arr` nodes via a **tag → constructor** dispatch.

### Serialization format (Option B)
Pre-order stream of u32s + an interned string table:
```
node  := tagId:u32  payload
payload depends on tag's shape:
  - fixed-arity ctor: each arg is a node (recurse) or a leaf
  - List<X> arg: count:u32 then that many nodes
  - leaf String: strId:u32 (index into a string table appended after the tree)
  - leaf Number: encoded literal
  - Loc: 6×u32 (start/end line/col/char)  (or a srcloc-id)
  - Option<X>: 0 (none) | 1 node (some)
```
`tagId` indexes a table generated from `ast.arr`'s variant order (the same 225-variant
order the seed's `dataTypeRanges` already tracks), so JS and Pyret agree on tags.

## The tag → constructor table
`ast.arr` has ~225 constructors. The deserializer needs `tagId -> ast.<ctor>` + the
arg-shape (which args are nodes / Lists / leaves / Loc / Option). This table can be
GENERATED by scanning `ast.arr`'s `data` decls (variant name + member annotations:
`:: Expr`→node, `:: List<...>`→list, `:: String`/`Number`→leaf, `:: Loc`→loc,
`:: Option<...>`→option). A build script emitting the table (JS side) + the Pyret
dispatch keeps both in sync.

## Recommended path
1. Option B host import + retargeted parse-pyret.js (smallest bridge to a RUNNING
   front-end on real source).
2. `parse-from-tree.arr` deserializer (skeleton in this dir) grown to all 225 ctors via
   the generated table.
3. Later, Option A (pure Pyret port) if we want zero JS in the parse step — but the JS
   GLR parser stays regardless (a Pyret GLR parser is a separate, much larger effort).

## First step taken
`self-host/parse-from-tree.arr` — a skeleton `surface-parse`/`from-tree` deserializer
for the core forms (program/block/num/str/id/op/app/let/if), showing the tag-dispatch +
recursive child-building shape. It parses cleanly (it's a sketch; the host import and the
full ctor table are TODO). `self-compiler/compiler/parse-pyret.arr` (the stub) should
later `include` this and call `from-tree(read-parse-tree())`.
